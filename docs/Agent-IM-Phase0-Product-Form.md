# Agent IM Phase 0 产品形态定义

> 用途：明确 `Phase 0` 阶段 Agent IM 应该以什么产品形态出现，以及为什么这样定义。  
> 目标：避免团队在“网页端 / App / CLI / 桌面客户端”之间反复摇摆，把有限资源集中到**最小可验证闭环**。

## 1. 一句话结论

**Phase 0 的主产品形态应定义为：桌面优先的 Web 控制台（Web Console）。**

配套形态建议为：

- `API / SDK`：平台底座，服务接入与联调
- `轻量 CLI`：开发者 / 节点辅助工具

不建议作为主形态的包括：

- 移动 App
- 原生桌面 App
- 纯命令行产品

---

## 2. 为什么是这个形态

Phase 0 当前要验证的不是“哪个端的体验最好”，而是以下问题：

1. 用户能否在一个会话里与一个或多个 Agent 协作
2. 平台能否完成最小可信调用闭环
3. 风险控制、审计、收据能否被用户和团队直观看见
4. demo 是否可稳定演示
5. 团队能否基于真实体验做 go/no-go 判断

这决定了 Phase 0 最适合的不是“消费级入口”，而是一个**信息密度高、便于操作、便于展示、便于验证**的控制台形态。

---

## 3. 推荐形态：桌面优先 Web 控制台

建议把 `Phase 0` 的主形态定义为：

**一个面向开发者型小团队 / 早期企业创新团队 / 内部演示与访谈场景的桌面 Web Console。**

它不是普通聊天网页，也不是官网，而是一个最小但完整的 Agent 协作控制台。

### 3.1 这个 Web Console 的核心职责

- 承载会话主闭环
- 承载 Agent 发现与加入
- 承载可信调用的可视化
- 承载高风险确认体验
- 承载审计与收据展示
- 承载 demo 与用户访谈

### 3.2 为什么必须是桌面优先

- 会话、目录、审计、风险信息密度较高，更适合桌面布局
- 便于在演示时同时展示消息流、参与者、风险状态、事件记录
- 更符合开发者型团队 / PM / 早期 B 端用户的使用习惯
- 前期无需处理移动端优先带来的复杂交互压缩

---

## 4. Phase 0 不推荐的主形态

## 4.1 不建议移动 App 作为主形态

原因：

- 当前核心不是随时随地使用，而是验证协作与可信调用闭环
- 风险、审计、目录等信息在移动端更难表达清楚
- 会额外引入登录态、多端同步、推送、安装分发等复杂度
- 会分散本应投入在平台闭环上的资源

结论：

**移动 App 可留待 Phase 1 或更后，不应成为 Phase 0 主形态。**

## 4.2 不建议原生桌面 App 作为主形态

原因：

- 当前尚不需要强调系统级权限、托盘常驻、原生文件集成
- 用 Web 控制台足以完成演示与验证
- 原生壳会增加工程与测试成本

结论：

**原生桌面客户端不是 Phase 0 必要项。**

## 4.3 不建议纯 CLI 作为主形态

原因：

- CLI 不利于展示“可信调用”的价值
- 不利于做用户访谈和业务判断
- 难以承载目录、风险提示、审计痕迹等可视化信息

结论：

**CLI 适合作为开发者辅助工具，不适合作为主产品。**

---

## 5. 推荐的三层产品结构

## 第一层：用户可见层

**桌面 Web 控制台**

这是 Phase 0 的主产品。

## 第二层：平台接入层

**API / SDK**

这是平台能力底座，用于：

- Agent 接入
- 会话映射
- 调用转发
- 审计与收据联动

## 第三层：工程辅助层

**轻量 CLI**

这是辅助工具，用于：

- 本地联调
- Agent 注册与调试
- 节点测试（若做节点）
- demo 脚本辅助

一句话：

**用户看 Web，开发接 API/SDK，工程用 CLI。**

---

## 6. Web 控制台的最小信息架构

Phase 0 不需要做大而全后台，只需要围绕主闭环搭建最小结构。

建议最小信息架构如下：

1. `会话页`
2. `目录页`
3. `审计 / 事件视图`
4. `高风险确认交互`

如果做节点，可增加：

5. `节点状态视图（可选）`

---

## 7. 必须有的页面 / 视图

## 7.1 会话页

必须支持：

- 创建 / 打开会话
- 查看消息流
- 发送消息
- 查看 Agent 返回
- 区分普通消息与系统/调用事件

这是整个产品的主容器。

## 7.2 目录页

必须支持：

- 查看 Agent 列表
- 查看 Agent 基础信息
- 看到来源标识
- 把 Agent 加入会话

这是“发现 -> 加入 -> 协作”的入口。

## 7.3 审计 / 事件视图

必须支持：

- 查看关键调用事件
- 查看是否被放行 / 限制 / 拒绝
- 查看关键收据

这是“可信调用”被用户感知到的核心位置。

## 7.4 高风险确认交互

必须支持：

- 至少一类高风险调用的明确确认
- 给出可理解的风险提示
- 让用户知道为什么这次不能默认自动执行

这不是“弹窗装饰”，而是产品差异化的核心体验。

## 7.5 节点状态视图（可选）

只有在 `Phase 0` 保留节点最小闭环时再做。

必须支持：

- 节点在线 / 离线
- 节点同步来的 Agent 可见

如果节点不做，这个视图可不做。

---

## 8. 页面布局建议

为了同时承载协作与可信调用，建议采用**桌面三栏或二栏扩展布局**。

建议结构：

- 左侧：会话列表 / 导航
- 中间：当前会话消息流
- 右侧：当前参与 Agent、来源、风险、调用事件、收据摘要

目录页可单独一页，也可作为抽屉 / 侧板进入。

原则不是“做得像 IM”，而是：

**做成最适合演示和验证可信调用的协作控制台。**

## 9. Web Console 与官网的关系

## 9.1 可以挂在官网体系下，但不应被设计成官网内容页

`Web Console` 完全可以放在官网同一套 Web 项目或同一域名体系内，但不建议把它理解为：

- 官网里的普通二级内容页
- 长滚动的营销页面
- 继续沿用官网导航、页脚、宣传式布局来承载产品使用

更准确的定义应是：

**同一站点中的两个区域：官网区（Marketing Zone） + 产品区（Product Zone）。**

其中：

- `官网区` 负责讲清楚是什么、为什么、案例、文档、CTA
- `产品区` 负责承载会话、目录、风险确认、审计、收据与 demo

一句话：

**Web Console 可以挂在官网体系下，但不应该被当作官网内容页来设计。**

## 9.2 Phase 0 推荐站点结构

最推荐的 Phase 0 落地方式是：

### 方案 A：同域不同路由

- `/`：官网首页
- `/product`：产品说明
- `/docs`：文档 / 开发者入口
- `/console`：Web Console
- `/console/conversations/:id`：会话详情
- `/console/agents`：Agent 目录
- `/console/audit`：审计 / 事件

这是最适合 Phase 0 的方案，因为：

- 开发与部署最快
- 品牌感统一
- 官网到产品体验的跳转最自然
- 适合演示、访谈与快速验证

### 方案 B：官网 + 产品子域

- `www.xxx.com`：官网
- `app.xxx.com`：Web Console

这个方案更适合 Phase 1 及之后，当产品边界、鉴权和多环境要求更清晰时再采用。

## 9.3 为什么不建议“官网内嵌式页面”

如果所谓“官网内嵌”指的是以下做法，则不建议：

- 把控制台做成一个普通 marketing 页面
- 用宣传页样式去包裹会话、目录、审计、风险确认
- 让产品页长期依附在官网布局中

原因：

- 官网和控制台的目标完全不同
- 控制台未来会越来越像产品，而不是页面
- 会影响登录态、权限、导航和信息架构
- 不利于表达“可信调用”这种高信息密度价值

## 9.4 推荐的页面关系

官网首页建议承担：

- 价值主张说明
- 可信调用叙事
- demo 截图 / 视频
- CTA：进入 Demo Console / 查看文档 / 申请试用

Web Console 建议承担：

- 创建会话
- 加入 Agent
- 发起任务
- 处理高风险确认
- 查看审计 / 收据

用户路径应是：

**先被官网教育，再进入 Console 体验。**

---

## 10. Phase 0 明确不做的形态能力

为了避免范围失控，以下能力不建议纳入 Phase 0：

- 移动 App 正式版
- 原生桌面客户端
- 多端一致性深度优化
- 复杂通知系统
- 大规模视觉 polish
- 消费级 onboarding 漏斗
- 硬件专属 UI

一句话：

**Phase 0 做“能验证”的控制台，不做“看起来完整”的多端产品。**

## 11. 对设计与前端的交接定义

## MVP Scope & Handoff Package

### MVP 核心价值主张

用户可以在一个桌面 Web 控制台中，与多个 Agent 协作，并清楚看到调用是否可信、为何被拦截、是否留下审计与收据。

### 必须实现的用户故事（按优先级）

1. 作为用户，我可以创建一个会话并发送一条消息。
2. 作为用户，我可以从目录中选择一个 Agent 加入会话。
3. 作为用户，我可以在会话中看到 Agent 返回结果。
4. 作为用户，我可以在高风险调用发生时看到明确提示，并完成一次确认。
5. 作为管理员或演示者，我可以看到关键调用事件与基础收据。

### 明确不做的功能清单

- 移动 App - 不属于当前最小验证闭环
- 原生桌面客户端 - 当前没有必要承担额外工程成本
- 复杂线程 / `@` / 多层消息交互 - 会分散主闭环
- 消费级精细视觉 polish - 不提升当前验证效率
- 硬件专属界面 - 硬件不在 Phase 0 交付范围内

### 成功指标

- **North Star Metric**: 每周可信调用完成数
- **辅助 KPI**:
  - 首次会话成功率
  - 高风险调用被正确拦截 / 升级比例
  - 审计事件覆盖率
  - 关键收据覆盖率

### 目标设备 / 平台

桌面优先 Web；默认面向笔记本与桌面浏览器。移动端只保证最基础可访问，不作为主体验设计目标。

### 核心用户旅程

1. 用户进入控制台并创建会话
2. 从目录中选择 Agent 加入会话
3. 用户输入任务并触发调用
4. 平台返回结果，并在必要时触发风险提示 / 确认
5. 用户查看调用事件、审计与收据

### 建议风格关键词

Professional, Trustworthy, Minimal, High-signal, Demo-ready

### 已知品牌 / 色调 / 语气要求

当前以“可信、克制、清晰、可验证”为主，不追求花哨与消费化情绪设计。

### 格式要求

全部用 Markdown，便于继续纳入现有文档体系。

---

## 12. 最后一句话

Phase 0 的产品形态不是 App，不是纯命令行，也不是成熟 SaaS 后台，而是：

**一个挂在官网体系下、但作为独立产品区存在的桌面优先最小 Web 控制台。**
