# Agent IM Sprint 1 任务清单

> 基于 `Agent-IM-Phase0-Sprint-Plan.md` 的 `Sprint 1：主闭环骨架` 拆解。  
> 目标：让团队可以直接据此开工、分工、跟踪。  
> Sprint 假设：**2 周**，核心目的是先证明“用户能把 Agent 拉进会话并看到结果”。

## 1. Sprint 目标

Sprint 1 结束时，必须能演示：

1. 创建一个会话
2. 从目录中选择一个 Agent 加入会话
3. 用户发送一条消息
4. 平台触发一次 A2A 调用
5. Agent 返回一条结果并回写到会话

如果做不到这一点，Sprint 1 视为未完成。

---

## 2. Sprint 1 明确不做

为了避免范围膨胀，本 Sprint 不做：

- 线程
- `@` 路由增强
- 复杂 UI polish
- 复杂目录筛选 / 推荐
- 多 Agent 并行编排
- 信任评分 / 信誉画像
- 节点接入
- 硬件接入

---

## 3. Sprint 1 Definition of Done

以下条件全部满足，才算 Sprint 1 Done：

- 会话基础数据模型落地
- 目录基础数据模型落地
- 至少一个 Agent 可注册并出现在目录中
- 至少一个 Agent 可被加入会话
- 用户消息可触发 A2A 请求
- A2A 响应可写回消息流
- 能跑通一条端到端演示路径

## 5. 详细任务拆解

## A. 对齐与验收准备

### S1-01 明确 Sprint 1 验收口径与 demo 脚本骨架

- **负责人**：PM
- **目标**：让团队从第一天就知道 Sprint 1 的唯一成功标准
- **产出**：
  - Sprint 1 验收 checklist
  - 最小 demo 步骤草稿
  - 不做清单确认版
- **完成标准**：
  - 团队确认 Sprint 1 只追一个闭环
  - 所有人知道最终要演示哪 5 个步骤

---

## B. 会话基础骨架

### S1-02 定义会话数据模型

- **负责人**：后端 / 平台
- **目标**：建立 `conversation` 最小对象
- **至少包含**：
  - id
  - title
  - state
  - created_at
  - updated_at
- **完成标准**：
  - 模型可被创建、读取

### S1-03 定义参与者数据模型

- **负责人**：后端 / 平台
- **目标**：支持 user / agent 作为统一参与者
- **至少包含**：
  - conversation_id
  - participant_type
  - participant_id
  - role
- **完成标准**：
  - 会话能挂载多个参与者

### S1-04 定义消息 / 事件数据模型

- **负责人**：后端 / 平台
- **目标**：统一承载用户消息、Agent 返回、系统事件
- **至少包含**：
  - conversation_id
  - sender_type
  - sender_id
  - content
  - event_type
  - created_at
- **完成标准**：
  - 会话可追加并读取消息流

### S1-05 创建会话 API

- **负责人**：后端 / 平台
- **完成标准**：
  - 用户可创建一个新会话并返回会话 ID

### S1-06 获取会话详情 API

- **负责人**：后端 / 平台
- **完成标准**：
  - 可返回会话基础信息、参与者列表、消息流

### S1-07 发送用户消息 API

- **负责人**：后端 / 平台
- **完成标准**：
  - 用户消息能进入会话消息流

---

## C. Agent 目录与加入会话

### S1-08 定义 Agent 基础模型

- **负责人**：后端 / 平台
- **目标**：建立目录最小对象
- **至少包含**：
  - id
  - name
  - description
  - source_type
  - status
- **完成标准**：
  - Agent 可在系统中注册并查询

### S1-09 支持 `agent_type` 与基础能力声明

- **负责人**：后端 / 平台
- **目标**：支持 `logical / execution`
- **至少包含**：
  - `agent_type`
  - 简单能力描述字段
- **完成标准**：
  - 目录中能区分 Agent 类型

### S1-10 Agent 注册 API

- **负责人**：后端 / 平台
- **完成标准**：
  - 至少可手动注册一个 Agent

### S1-11 Agent 列表 / 详情 API

- **负责人**：后端 / 平台
- **完成标准**：
  - 可查看 Agent 列表与单个详情

### S1-12 Agent 加入会话动作

- **负责人**：后端 / 平台
- **完成标准**：
  - 目录中的 Agent 能变成会话参与者

---

## D. A2A 最小调用链

### S1-13 A2A Gateway 最小封装

- **负责人**：协议 / 后端
- **目标**：对接至少一个可调用 Agent
- **完成标准**：
  - 给定 Agent 配置可发起一次请求并收到响应

### S1-14 会话消息映射为 A2A 请求

- **负责人**：协议 / 后端
- **目标**：用户消息可触发 Agent 调用
- **完成标准**：
  - 当会话中存在 Agent 参与者时，能发起一次调用

### S1-15 A2A 响应回写消息流

- **负责人**：协议 / 后端
- **目标**：让会话真正闭环
- **完成标准**：
  - Agent 返回结果能写成一条会话消息

---

## E. 最小可操作前端

### S1-16 最小会话页 UI

- **负责人**：前端
- **优先级**：P1
- **完成标准**：
  - 可创建会话
  - 可查看消息流
  - 可发送消息

### S1-17 最小目录页 UI

- **负责人**：前端
- **优先级**：P1
- **完成标准**：
  - 可查看 Agent 列表
  - 可将 Agent 加入会话

说明：

- 如果资源不足，可先用极简控制台形式，不要求 polished UI。

---

## F. 验证与收尾

### S1-18 端到端 demo 路径验证

- **负责人**：QA / 全员
- **目标**：把 Sprint 1 做成一条可复现路径
- **完成标准**：
  - 按 demo 脚本能稳定跑通至少 3 次

### S1-19 Sprint 回顾与下一 Sprint 输入整理

- **负责人**：PM
- **目标**：让 Sprint 2 能直接开工
- **产出**：
  - Sprint 1 结果总结
  - 阻塞项清单
  - Sprint 2 可信调用闭环输入清单
- **完成标准**：
  - 团队对 Sprint 2 范围无重大歧义

## 7. 建议站会跟踪字段

每个任务建议至少跟踪：

- 状态：todo / doing / blocked / done
- 阻塞项
- 是否影响 Sprint 退出标准
- 负责人
- 预计完成时间

特别要标记：

- 哪些任务阻塞 `S1-18 端到端 demo`

---

## 8. Sprint 1 Kill Criteria

若到 Sprint 1 末仍出现以下情况之一，应考虑立即收缩范围，而不是继续补功能：

1. 仍无法稳定创建会话并看到 Agent 返回
2. 目录无法支撑加入会话动作
3. A2A 调用链无法稳定跑通
4. 团队为了 demo 不得不手动替代关键链路

---

## 9. 下一步如何衔接 Sprint 2

Sprint 1 完成后，Sprint 2 不应重开新战线，而应直接接住以下产出：

- 会话消息流
- Agent 身份基础结构
- 最小能力声明
- A2A 调用链

然后只补：

- 高风险标记
- Trust Policy v0
- 审计事件
- 基础收据

---

## 10. 一句话总结

Sprint 1 的任务清单不是为了“把架子搭漂亮”，而是为了最快回答一句话：

**用户能不能把 Agent 拉进一个会话，并看到它真实完成一次调用。**
