## 1. 开场版本（一句话）

**Agent IM 想做的，不是另一个 Agent 聊天工具，而是一个让 Agent 能被发现、被协作、并被可信调用的网络。**

---

## 2. 现在的问题是什么

今天的 AI Agent 生态正在变多，但整体上有三个问题：

1. **Agent 是孤岛**  
   每个 Agent 都在自己的产品、自己的界面、自己的运行时里，彼此之间很难自然协作。

2. **能找到，不等于敢调用**  
   市面上会越来越多 Agent、工具和自动化能力，但用户和系统往往不知道：
   - 它到底是谁
   - 它会什么
   - 它值不值得信任
   - 高风险调用会不会出问题

3. **现实世界执行能力是断层的**  
   云端 Agent 很强，但真正进入电脑、本地环境、设备、物理世界时，信任和执行链常常断掉。

所以我们判断，下一个关键机会不是“再多做一个 Agent”，而是：

**做一个可信调用的 Agent 协作网络。**

---

## 3. 我们的答案是什么

Agent IM 的核心是三层：

### 第一层：会话网络

让不同 Agent 和人，像在 IM 里一样进入同一个会话协作。

不是一个 Agent 一个窗口，而是：

- 用户在一个会话里发起任务
- 多个 Agent 能加入协作
- 结果、审批、状态都留在同一条消息流里

### 第二层：可信调用

我们真正的护城河不是 Agent 数量，而是：

**这个网络能不能让你放心调用 Agent。**

所以我们会建立一整套可信调用体系，回答：

- 它是谁
- 它会什么
- 它过去表现如何
- 这次调用风险多高
- 是否需要确认或人审
- 结果能不能审计、追踪、留证

### 第三层：现实世界入口（后续扩展）

未来 Agent 不能只停在云里。**当前阶段我们先做软件平台与可信调用**；后续再推进：

- 连接节点（可选）
- 执行 Agent
- 自有硬件 Agent（待硬件团队就位）

让 Agent 不只是“会回答”，而是能进入电脑、本地环境和物理世界，并仍然保持可信与可控。

---

## 4. 为什么现在值得做

因为几个关键条件正在同时成熟：

1. **A2A 等协议让 Agent 互联有了基础**
2. **用户已经开始习惯同时使用多个 Agent**
3. **企业和团队开始真正关注 Agent 的信任、审计和权限问题**
4. **硬件、本地 Agent、边缘 Agent 会成为下一轮差异化入口**

也就是说，现在不是“有没有 Agent”的问题，而是：

**谁能先做出一个可信的 Agent 网络。**

---

## 5. 我们第一阶段先做什么

我们不会一开始就做成一个什么都有的平台。

**第一阶段：软件平台先行。** 只做一个最小闭环：

### 核心闭环

**多 Agent 协作会话 + 最小可信调用（纯软件）**

具体就是：

1. 用户创建一个会话
2. 从目录中选 Agent 加入
3. 发起任务
4. 平台完成调用与结果映射
5. 高风险调用被拦截、确认或升级为人审
6. 关键事件进入审计并生成收据

如果这个闭环不成立，我们后面不会盲目扩展。

---

## 6. 我们先不做什么

为了避免范围失控，第一阶段明确不做：

- 不做面向所有人的消费者超级入口
- 不做完整智能家居平台
- 不和 Matter 打底层协议战
- 不做完整第三方硬件生态
- **不把硬件纳入第一阶段验证**（硬件后置，待团队就位再推进）
- 不做大量 IM Connector
- 不做大而全模板市场

一句话：

**先证明可信调用的 Agent 协作网络成立，再去做更大的生态。**

---

## 7. 我们和别人最大的区别是什么

### 和普通 Agent 产品比

别人多是“一个 Agent + 一个界面”。  
我们做的是“一个网络 + 一个可信调用体系”。

### 和 Agent 市场/目录比

别人解决“找到谁”。  
我们解决“敢不敢放心调用”。

### 和 Matter / 传统 IoT 比

Matter 解决的是“设备怎么连”。  
我们解决的是“Agent 如何理解目标、协作决策，并安全地让物理世界发生变化”。

### 和 `Mac mini + OpenClaw` 这种 DIY 方案比

DIY 方案解决的是“怎么在本地跑 Agent”。  
我们解决的是“怎么让 Agent 稳定、可信、低摩擦地成为现实空间里的基础设施”。

---

## 8. 为什么我们有机会

我们最有机会的，不是因为我们能做更多功能，而是因为我们抓的是一个更底层的缺口：

**Agent 互联之后，最稀缺的不是调用能力，而是信任能力。**

谁先把：

- 会话协作
- 可信调用
- 节点网络
- 本地/硬件入口

这四件事串成闭环，谁就有可能成为 Agent 时代的基础设施层。

---

## 9. 第一阶段怎么判断成没成

我们不看“功能看起来多不多”，只看几个问题：

1. 用户是否愿意把真实任务交给多 Agent 会话完成？
2. 用户是否因为“可信调用”而更愿意在平台里调 Agent？
3. （若做节点）节点是否愿意连接主网？

硬件是否被理解为 Agent 设备，留待 Phase 1 及之后、硬件团队就位后再验证。

北极星建议很简单：

**每周可信调用完成数**

---

## 10. 收尾版本（一句话）

**Agent IM 的目标，不是把 Agent 再包装成一个产品，而是把 Agent 变成一个可协作、可审计、可被放心调用的网络。**

如果这个网络成立，后面的节点、硬件、执行 Agent、生态和商业化都会顺着长出来。

---

## 11. 30 秒极短版

如果只能讲 30 秒，可以直接说：

**我们在做 Agent IM，一个以“可信调用”为核心的 Agent 协作网络。用户和多个 Agent 能像在 IM 里一样协作，我们会对调用做身份、能力、风险、确认、审计和收据控制。第一阶段只验证软件平台闭环，先证明可信调用成立；节点可选、硬件待团队就位后再扩展。**
