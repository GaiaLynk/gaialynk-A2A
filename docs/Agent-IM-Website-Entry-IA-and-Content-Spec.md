## 2. 页面职责与主 CTA

## 2.1 首页

职责：建立统一认知。  
主 CTA：

- `Start Building`
- `Book a Demo`

必须包含：

- 一句话定位
- 三段核心价值（协作、可信、可扩展）
- 可信证据区（审计/收据/风险控制）
- 快速入口（开发者 / 团队）

## 2.2 开发者入口页

职责：降低接入门槛。  
主 CTA：`Read Quickstart`

必须包含：

- 5 分钟接入路径
- API/SDK 入口
- 示例与常见问题
- OSS 与托管边界说明

## 2.3 信任与安全页

职责：把“可信调用”讲清楚。  
主 CTA：`See Trust Flow`

必须包含：

- 风险等级与决策机制
- 审计链路与收据验证
- 高风险确认/人审机制
- 治理与可追溯表达

## 2.4 场景页

职责：把价值落地到真实任务。  
主 CTA：`Try This Workflow`

建议首批 2-3 个场景：

- 多 Agent 开发协作
- 高风险动作审批流程
- 节点接入与跨边界协作

---

## 3. 内容层级模板（每页通用）

1. 问题（用户当前痛点）  
2. 方案（我们如何解决）  
3. 证据（演示、指标、可验证机制）  
4. 动作（下一步 CTA）  

禁止：

- 无证据口号
- 多主线叙事
- 页面无主动作

---

## 4. 文案风格规范

- 简短、具体、可验证
- 避免“革命性/颠覆式”等空泛词
- 同一概念三语术语一致
- 文案承诺不得超出现有交付能力

术语建议统一：

- Trustworthy Invocation -> 可信调用 / 可信調用
- Review Queue -> 待确认队列 / 待確認佇列
- Receipt -> 收据 / 收據
- Audit Trail -> 审计链路 / 稽核鏈路

---

## 5. 设计与体验规范（简版）

- 首屏聚焦 1 个核心结论 + 2 个动作
- 信息密度分层，优先可读性
- 动效仅用于引导，不用于炫技
- 对比度与可访问性达标
- 移动端优先检查关键路径

---

## 6. 发布前核对表

- 页面结构符合 IA
- 所有 CTA 可达且目标页面存在
- 三语切换无死链与内容错位
- 关键术语一致
- 页面内容与产品真实状态一致
