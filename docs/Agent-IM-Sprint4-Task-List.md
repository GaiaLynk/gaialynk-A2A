# Agent IM Sprint 4 任务清单

> 基于 `Agent-IM-Phase0-Sprint-Plan.md` 的 `Sprint 4：Demo、可视化与验证` 拆解。  
> 目标：把前 3 个 Sprint 的能力做成一个“能演示、能验证、能判断去留”的 Phase 0 成果。  
> Sprint 假设：**2 周**，核心不是继续扩功能，而是把结果做成可判断的产品证据。

## 关联文档

- 总计划：`Agent-IM-Phase0-Sprint-Plan.md`
- 上一阶段：`Agent-IM-Sprint3-Task-List.md`
- 角色分工：`Agent-IM-Phase0-Owner-Mapping.md`
- 上游 backlog：`Agent-IM-Phase0-Backlog.md`

---

## 1. Sprint 目标

Sprint 4 结束时，必须同时完成：

1. 端到端 demo 可稳定演示
2. 团队可看到关键指标数据
3. 可执行至少 3～5 位目标用户访谈
4. 能基于验收表判断是否进入 Phase 1

如果做不到这一点，Sprint 4 视为未完成。

---

## 2. Sprint 4 明确不做

为了避免“临门一脚又开新坑”，本 Sprint 不做：

- 新增底层协议大战线
- 新增复杂编排能力
- 新增复杂硬件控制
- 新增复杂节点治理
- 完整 UI 重构
- 额外消费者功能
- 额外商业化功能

一句话：**只做演示、验证、可视化与 go/no-go 判断。**

---

## 3. Sprint 4 Definition of Done

以下条件全部满足，才算 Sprint 4 Done：

- 存在一条稳定的端到端 demo 路径
- 最小控制台可支撑演示
- 来源 / 风险 / 审计有最小可见性；（若做节点）节点状态有最小可见性
- 北极星及关键辅助指标可被统计
- 目标用户访谈可执行并形成结论
- 团队可据此明确进入 Phase 1 或收缩方向

## 5. 详细任务拆解

## A. 目标与决策口径

### S4-01 明确 Sprint 4 验收口径与 go/no-go 标准

- **负责人**：PM
- **目标**：让 Sprint 4 不是“做收尾”，而是“做判断”
- **产出**：
  - Sprint 4 验收标准
  - go / no-go 判断口径
  - 访谈要验证的问题清单
- **完成标准**：
  - 团队明确知道 Sprint 4 要回答什么问题

---

## B. 最小控制台与可视化

### S4-02 最小会话页 UI 完成可演示版

- **负责人**：前端
- **目标**：支撑演示会话主闭环
- **完成标准**：
  - 可创建 / 打开会话
  - 可查看消息流
  - 可发送消息

### S4-03 最小目录页 UI 完成可演示版

- **负责人**：前端
- **完成标准**：
  - 可查看 Agent 列表
  - 可把 Agent 加入会话

### S4-04 审计事件最小视图

- **负责人**：前端 + 后端
- **目标**：让“可信调用”可见
- **完成标准**：
  - 至少能看到关键调用事件

### S4-05 来源与风险标签最小展示

- **负责人**：前端
- **完成标准**：
  - Agent 来源可见
  - 高风险标签可见

### S4-06 节点状态最小展示（若做节点）

- **负责人**：前端
- **完成标准**：
  - （若做节点）节点在线状态可见
  - 不要求硬件状态（硬件后置）

### S4-07 高风险确认交互补齐

- **负责人**：前端 + PM
- **完成标准**：
  - 至少一类高风险调用有清晰确认体验

---

## C. Demo 与环境准备

### S4-08 端到端 demo 脚本定稿

- **负责人**：PM + QA
- **目标**：让外部演示和内部验收都用同一套路径
- **建议至少包含**：
  - 主网用户创建会话
  - 加入 Agent
  - 发起普通调用
  - 发起高风险调用
  - 展示审计 / 收据
  - （若做节点）展示节点在线状态
- **完成标准**：
  - demo 步骤文档化

### S4-09 demo 环境与演示数据准备

- **负责人**：QA + 后端 / 前端
- **完成标准**：
  - 演示账号、演示 Agent 可复用；（若做节点）演示节点可复用

---

## D. 埋点、验收、访谈

### S4-10 北极星与关键埋点落地

- **负责人**：后端 / 平台 + PM
- **目标**：至少能统计 `每周可信调用完成数`
- **至少覆盖**：
  - 会话创建
  - Agent 加入会话
  - 调用发起
  - 调用结果
  - 高风险调用拦截/确认
  - 收据生成
- **完成标准**：
  - 可拉出最小指标结果

### S4-11 验收 checklist 定稿

- **负责人**：PM + QA
- **完成标准**：
  - 团队用统一 checklist 做验收

### S4-12 目标用户访谈执行

- **负责人**：PM
- **目标**：获取 go/no-go 证据
- **建议重点问题**：
  - 你是否理解这是“可信调用的 Agent 协作网络”？
  - 你是否觉得高风险控制有价值？
  - （若做节点）节点是否增强了你对产品的理解？
  - 你会不会用它做真实任务？
- **完成标准**：
  - 至少完成 3～5 位目标用户访谈

### S4-13 结果总结：Phase 1 进入建议

- **负责人**：PM
- **产出**：
  - Phase 0 结果总结
  - 是否进入 Phase 1 的建议
  - 如果不进入，建议收缩项
- **完成标准**：
  - 团队可以据此做下一阶段决策

## 8. Sprint 4 Kill Criteria

若到 Sprint 4 末仍出现以下情况之一，应考虑先收缩而不是直接进入 Phase 1：

1. 端到端 demo 仍无法稳定演示
2. 用户无法理解“可信调用”与普通 Agent 工具的区别
3. 高风险调用控制虽然存在，但用户感知不到价值
4. 指标无法产出，无法判断方向是否成立
5. 团队只能继续依赖新增功能来“解释项目”，而不能靠现有闭环证明价值

---

## 9. 下一步如何衔接 Phase 1

如果 Sprint 4 结论积极，Phase 1 建议优先放大：

- 多维信誉画像
- 更完整的 Trust Engine
- 基础编排与意图路由
- 节点可用化
- **硬件接入与本地行动中枢**（待硬件团队就位后推进）

如果 Sprint 4 结论不积极，则应优先收缩：

- 用户群
- 节点范围
- 可信调用边界

---

## 10. 一句话总结

Sprint 4 的任务清单不是为了“把产品做漂亮”，而是为了最终回答一句话：

**这个“可信调用的 Agent 协作网络”到底值不值得继续放大。**
