# Agent IM Phase 0 Owner Mapping

> 这不是一份“正式组织架构文档”，而是一份 **Phase 0 执行期的轻量责任映射表**。  
> 目的只有一个：让 `PRD -> Backlog -> Sprint -> 验收` 这条链路里，每块事情都有人主责，不出现“大家都知道，但没人真正推进”的灰区。

## 1. 使用原则

这份文档建议这样用：

- 按**角色能力域**挂主责，而不是按职级挂主责
- 每块事情只设 **一个 DRI**
- 允许协作，但不允许没有最终 owner
- Sprint 里看的是任务 owner；Phase 0 里看的是能力域 owner
- 如果团队很小，一个人可以同时覆盖多个角色，但责任边界仍然要清楚

一句话：**可以人少，但不能角色失焦。**

---

## 2. 建议角色映射

| 角色 | 核心职责 | Phase 0 重点交付 |
|------|----------|------------------|
| **产品 / PM** | 范围收敛、优先级、验收标准、访谈与决策 | PRD、Backlog、Sprint 验收、go/no-go 结论 |
| **后端 / 平台** | 核心服务、数据模型、API、审计、收据、Trust Policy | 会话、目录、调用、审计、收据、信任决策最小闭环 |
| **前端 / 客户端** | 用户操作入口与可视化 | 会话页、目录页、风险提示、审计视图、控制台 |
| **协议 / 节点** | Node–Hub 接入、注册、心跳、目录同步（若做节点） | 节点注册、心跳、最小目录同步 |
| **设计 / UX** | 演示级体验、确认交互、风险表达 | 最小控制台体验、确认流程 |
| **测试 / QA** | 联合验证、回归、验收、demo 稳定性 | 端到端验证、验收 checklist、demo 环境管理 |

**硬件 / 固件**：Phase 0 不设主责；待 Phase 1 及之后硬件团队就位后再纳入。

---

## 3. Phase 0 各模块建议 DRI

| 模块 | 建议 DRI | 关键协作角色 | 备注 |
|------|----------|--------------|------|
| `PRD / Backlog / Sprint 节奏` | 产品 / PM | 全员 | PM 必须主导，不宜分散 |
| `会话主闭环` | 后端 / 平台 | 前端、PM | 这是主线，不要多 owner |
| `目录与 Agent 接入` | 后端 / 平台 | 前端、协议 / 节点 | 数据模型统一很关键 |
| `A2A / 调用闭环` | 后端 / 平台 | PM、QA | 先闭环，后扩展 |
| `Trust Policy 最小版` | 后端 / 平台 | PM、前端 | 规则解释权需要 PM 参与 |
| `审计 / 收据` | 后端 / 平台 | 前端、QA | 直接影响“可信调用”可验证性 |
| `节点接入`（若做） | 协议 / 节点 | 后端 / 平台、QA | 协议与平台边界要早定 |
| `最小控制台` | 前端 / 客户端 | 设计 / UX、PM | 为 demo 与验证服务 |
| `高风险确认体验` | 设计 / UX | 前端、PM | 表达准确比炫更重要 |
| `demo 与验收` | 测试 / QA | PM、前端、后端 | QA 要有明确主责 |
| `用户访谈与结论` | 产品 / PM | 设计 / UX | 不能交给纯研发代替 |

**`硬件接入`**：Phase 0 不设 DRI；后置至 Phase 1 及之后。

## 4. 按 Sprint 的主责分布建议

## Sprint 1：主闭环骨架

建议主责：

- `PM`：验收标准、用户故事、demo 主路径
- `后端 / 平台`：会话模型、消息流、最小目录模型
- `前端 / 客户端`：会话最小入口
- `QA`：主闭环回归验证

## Sprint 2：可信调用最小闭环

建议主责：

- `PM`：风险分级口径、确认边界
- `后端 / 平台`：Trust Policy、审计、收据、调用状态
- `前端 / 客户端`：来源 / 风险展示、高风险确认
- `QA`：普通 / 高风险调用分路径验证

## Sprint 3：网络最小证明与控制台增强

建议主责：

- `协议 / 节点`（若做节点）：节点注册、心跳、目录同步
- `后端 / 平台`：节点模型与接收逻辑（若做节点）
- `前端 / 客户端`：最小控制台、来源/风险/审计视图
- `设计 / UX`：高风险确认体验
- `PM`：验收口径、demo 骨架
- `QA`：端到端 demo 验证（软件 + 可选节点）

**硬件不纳入本 Sprint。**

## Sprint 4：Demo、可视化与验证

建议主责：

- `前端 / 客户端`：最小控制台
- `设计 / UX`：高风险确认体验与演示体验
- `QA`：demo 稳定性、验收 checklist
- `PM`：访谈、指标解读、go/no-go 建议
- `后端 / 平台`：指标埋点与基础数据

## 5. RACI 轻量建议

如果你们不想上完整项目管理系统，最小可以按下面规则跑：

- `A`：最终拍板人，通常是 PM
- `R`：实际主做的人，通常是模块 DRI
- `C`：关键协作者
- `I`：需要同步的人

建议只在**跨角色任务**上写 RACI，例如：

| 任务类型 | A | R | C | I |
|----------|---|---|---|---|
| 风险分级口径定义 | PM | PM | 后端、前端、QA | 全员 |
| Trust Policy 接口落地 | PM | 后端 / 平台 | 前端、QA | 全员 |
| 高风险确认交互 | PM | 前端 / 设计 | 后端、QA | 全员 |
| 节点目录同步（若做节点） | PM | 协议 / 节点 | 后端、QA | 全员 |
| 端到端 demo 验收 | PM | QA | 全员 | 全员 |

**硬件首次绑定**：Phase 0 不适用；待 Phase 1 及之后再设 RACI。

## 6. Phase 0 最容易失焦的责任灰区

最容易出问题的，不是没人做，而是这些地方容易“多人参与但没人真正负责”：

1. `Trust Policy` 的规则定义  
如果 PM 不主导，容易变成纯技术实现；如果后端不主导，又容易停留在 PPT。

2. `高风险确认体验`  
如果只交给前端，容易做成弹窗；如果只交给设计，容易脱离真实流程。

3. `节点协议边界`  
如果平台和节点团队不先定边界，后续联调会很痛。（若不做节点，本项不适用。）

4. `demo 与访谈`  
如果没有 QA 和 PM 联合主责，最后经常会出现“功能有了，但无法稳定展示，也无法产出判断”。

**硬件首次绑定**：Phase 0 不纳入；待硬件团队就位后在后续阶段再定义责任边界。

---

## 7. 适合你们当前阶段的最小组织建议

如果团队很小，建议最小按下面方式覆盖：

- `PM / Founder`
- `平台后端`
- `前端`
- `协议 / 节点 or 全栈`（若做节点）
- `设计 / UX`
- `QA / 兼项目运营`

**硬件 / 固件**：Phase 0 不要求；待后续阶段再纳入。

如果人更少，也尽量别少掉这三类能力：

- `PM 判断`
- `平台实现`
- `QA / 验证`

因为 Phase 0 不是拼功能数，而是拼**收敛判断能力**。

---

## 8. 建议怎么落地到现有文档体系

最实用的接法不是再开一套管理体系，而是直接这样挂：

- `PRD`：写范围与验收
- `Backlog`：写优先级与依赖
- `Sprint Task List`：写任务 owner
- `Owner Mapping`：只保留能力域 DRI

也就是说：

**这份文档负责回答“这类事情总体谁主责”；**

**Sprint 任务清单负责回答“这个具体任务这周谁来做”。**

---

## 9. 一句话总结

Phase 0 的 owner mapping 不需要很重，但必须足够清楚：

**会话、可信调用、控制台、demo、访谈必须有人扛结果；节点（若做）有人扛；硬件后置，不占 Phase 0 主责。**
