## 1. 使用原则

### 1.1 优先级定义

- **P0-Must**：Phase 0 不做完，这一阶段就不算成立
- **P1-Should**：强烈建议做，能显著提升闭环质量，但不应阻塞主闭环
- **P2-Could**：有余力再做，不能挤占 P0 / P1 资源

### 1.2 判断原则

每个 backlog 项都必须回答两个问题：

1. 它是否直接增强“可信调用的 Agent 协作闭环”？
2. 如果不做它，Phase 0 的核心 demo 是否还能成立？

如果两个答案分别是“否 / 能”，则优先级不应高于 `P2-Could`。

---

## 2. Phase 0 总体交付物

Phase 0 结束时，应至少交付（**软件平台先行**）：

1. 主网最小可用闭环
2. 最小目录与 Agent 接入
3. 最小可信调用与审计
4. （可选）Node–Hub 最小接入
5. 一条端到端 demo 路径

硬件最小接入**后置**至 Phase 1 及之后，见下文「后续扩展 / Deferred」。

## 4. 详细 Backlog

## E1. 会话与消息闭环

### 目标

让用户、Agent 能在一条会话流中协作，形成产品主容器。

### 子项

| ID | 事项 | 优先级 | 依赖 | 产出/说明 | 验收标准 |
|----|------|--------|------|-----------|----------|
| E1-1 | `Conversation` 数据模型 | P0-Must | 无 | 会话表、基础状态字段 | 能创建/读取会话 |
| E1-2 | `Participant` 数据模型 | P0-Must | E1-1 | 支持 user / agent 参与者 | 会话可显示多参与方 |
| E1-3 | `Message/Event` 数据模型 | P0-Must | E1-1 | 支持消息流与系统事件 | 消息能按时间顺序展示 |
| E1-4 | 创建会话 API | P0-Must | E1-1 | 基础 CRUD 入口 | 用户可创建一个新会话 |
| E1-5 | 会话详情 API | P0-Must | E1-1,E1-3 | 返回消息流与参与者 | 会话页可加载 |
| E1-6 | 会话消息追加 API | P0-Must | E1-3 | 支持用户消息进入流 | 用户可发送一条消息 |
| E1-7 | 基础会话页 / 控制台视图 | P1-Should | E1-4,E1-5,E1-6 | 最小可操作 UI | 不依赖脚本即可演示基本流 |

### 备注

- `Thread`、`@`、在线状态增强都不在 Phase 0 主范围。

---

## E2. Agent 目录与接入

### 目标

让用户可以发现 Agent，并把 Agent 拉入会话。

### 子项

| ID | 事项 | 优先级 | 依赖 | 产出/说明 | 验收标准 |
|----|------|--------|------|-----------|----------|
| E2-1 | Agent 基础模型 | P0-Must | 无 | 名称、来源、agent_type、描述 | Agent 可被存储 |
| E2-2 | `agent_type` 支持 | P0-Must | E2-1 | `logical / execution` | 目录中可区分类型 |
| E2-3 | 基础能力声明字段 | P0-Must | E2-1 | 至少支持文本描述 + 简单能力结构 | Agent 详情可展示能力摘要 |
| E2-4 | Agent 注册 API | P0-Must | E2-1 | 平台录入/注册入口 | 可注册一个 Agent |
| E2-5 | Agent 列表 / 详情 API | P0-Must | E2-1 | 最小目录查询 | 用户能看到 Agent 列表 |
| E2-6 | “加入会话”动作 | P0-Must | E1-2,E2-5 | Agent 变成会话参与者 | Agent 能出现在会话中 |
| E2-7 | 来源标识展示 | P1-Should | E2-1 | 官方托管 / 自建 / 节点 / 厂商 | 目录页可区分来源 |

### 备注

- Phase 0 不做复杂筛选、推荐、能力匹配。

---

## E3. A2A 调用映射

### 目标

让平台不只是“展示 Agent”，而是真能发起调用并收回结果。

### 子项

| ID | 事项 | 优先级 | 依赖 | 产出/说明 | 验收标准 |
|----|------|--------|------|-----------|----------|
| E3-1 | A2A Gateway 最小封装 | P0-Must | E2-4 | 对接外部 Agent 请求/响应 | 能请求一个外部 Agent |
| E3-2 | 会话消息 -> A2A 请求映射 | P0-Must | E1-6,E3-1 | 用户消息可触发 Agent 调用 | 发消息可触发请求 |
| E3-3 | A2A 响应 -> 消息回写 | P0-Must | E1-3,E3-1 | Agent 结果写回会话 | 用户能看到 Agent 返回 |
| E3-4 | 调用失败处理 | P1-Should | E3-3 | 最小错误态与事件记录 | 失败时会话中可见错误信息 |
| E3-5 | 多 Agent 同会话支持 | P1-Should | E2-6,E3-3 | 至少支持顺序调用两个 Agent | Demo 中能拉两个 Agent |

### 备注

- Phase 0 不要求复杂并行编排。

---

## E4. 最小可信调用与审计

### 目标

验证“这个网络不仅能调 Agent，而且能可信地调 Agent”。

### 子项

| ID | 事项 | 优先级 | 依赖 | 产出/说明 | 验收标准 |
|----|------|--------|------|-----------|----------|
| E4-1 | Agent 稳定身份字段 | P0-Must | E2-1 | Agent ID、来源、主体最小结构 | 每个 Agent 有稳定身份 |
| E4-2 | 高风险能力标识字段 | P0-Must | E2-3 | 能力至少支持 low/high 基础风险等级 | 高风险能力可被识别 |
| E4-3 | 最小 Trust Policy v0 | P0-Must | E4-1,E4-2 | 未验证身份不放行；高风险不自动执行 | 至少一类调用会被拦截/升级 |
| E4-4 | 审计事件模型 | P0-Must | E1-3,E3-2 | 关键调用写入 audit_event | 可查看关键调用事件 |
| E4-5 | 基础收据模型 | P0-Must | E4-4 | 关键事件生成 receipt | 可展示一个关键收据 |
| E4-6 | 高风险确认交互 | P1-Should | E4-3 | 至少一种确认机制 | 可演示高风险调用确认 |
| E4-7 | 来源/风险提示展示 | P1-Should | E2-7,E4-2 | UI 或控制台可见 | 用户知道为什么是高风险 |

### Phase 0 推荐硬规则

- 身份不可验证：拒绝
- 能力未声明：拒绝
- 高风险能力：不自动执行
- 关键调用：必须写审计

## E5. Node–Hub 最小闭环（可选）

### 目标

证明主网不是单机产品，而是网络的起点；Phase 0 作为可选软件网络雏形证明。

### 子项

| ID | 事项 | 优先级 | 依赖 | 产出/说明 | 验收标准 |
|----|------|--------|------|-----------|----------|
| E5-1 | 节点注册接口 | P1-Should | 无 | 节点拿到 `node_id` | 一个节点可注册成功 |
| E5-2 | 心跳机制 | P1-Should | E5-1 | 节点在线状态维护 | 节点可被识别在线/离线 |
| E5-3 | 节点基础模型 | P1-Should | E5-1 | 节点来源、状态、能力概览 | 节点可被平台识别 |
| E5-4 | 目录同步 MVP | P1-Should | E2-1,E5-1 | 节点 Agent 可同步到主网 | 主网目录中可看到节点 Agent |
| E5-5 | 节点状态可视化 | P2-Could | E5-2 | 控制台最小展示 | 演示时能看见节点在线状态 |

### 备注

- Phase 0 不强求跨节点消息中继完整闭环。
- 若资源紧张，可整体延后至 Phase 1。

## E6. 硬件最小闭环（后续扩展 / Deferred）

### 目标

### 子项（Deferred）

| ID | 事项 | 说明 |
|----|------|------|
| E6-1 | 首次绑定流程 | 扫码/短码/设备绑定流程 |
| E6-2 | 设备注册模型 | 设备 ID、用户归属、在线状态 |
| E6-3 | 在线状态上报 | 设备 presence 最小模型 |
| E6-4 | 基础会话接入 | 设备能进入会话体系 |
| E6-5 | 状态反馈机制 | 指示灯 / 小屏 / 状态页至少一种 |
| E6-6 | 品类验证访谈 | 3～5 位用户访谈 |

### 备注

- Phase 0 不做本地行动中枢、复杂设备控制；整体后置。

## 5. 建议执行顺序

## Wave 1：主闭环骨架

- E1 会话与消息闭环
- E2 Agent 目录与接入
- E3 A2A 调用映射

**目标**：先证明“能协作”。

## Wave 2：可信调用骨架

- E4 最小可信调用与审计

**目标**：再证明“能可信地协作”。

## Wave 3：网络证明（可选）与可演示化

- E5 Node–Hub 最小闭环（若做）
- E7 基础控制台 / 极简 UI
- E8 来源/风险可视化与高风险确认（部分可在 Sprint 2 已做）

**目标**：证明这不只是单机产品（若做节点）；并让产品可演示、可感知。

## Wave 4：Demo 与验证

- E9 Demo、埋点与验证

**目标**：让产品能被演示、被理解、被验证、能判断是否进入 Phase 1。

**说明**：硬件闭环（E6）不在 Phase 0；待硬件团队就位后在后续阶段推进。

---

## 6. 资源分配建议

若资源有限，建议按以下顺序保资源：

1. **平台主闭环**：E1 + E2 + E3 + E4
2. **演示与验证**：E9
3. **UI 与可感知**：E7 + E8
4. **网络证明（可选）**：E5

硬件（E6）不占用 Phase 0 资源；待团队就位后单独排期。

如果资源紧张，宁可：

- UI 简陋
- 节点不做或延后

也不要牺牲：

- 调用闭环
- 高风险拦截
- 审计与收据

---

## 7. 建议的 Phase 0 Kill Criteria

以下任一情况出现，应考虑及时收缩而不是继续加功能：

1. 用户不认为“可信调用”有额外价值
2. 多 Agent 会话没有比单 Agent 使用方式更有吸引力
3. 端到端 demo 无法稳定演示、无法据此做 go/no-go 判断
4. 团队需要不断引入 P2-Could 项目才能让 demo 看起来成立

（若做节点）节点运营者对连接主网无兴趣，可视为节点轨道的收缩信号，不阻塞软件平台主闭环判断。硬件品类验证后置，不纳入 Phase 0 Kill Criteria。

---

## 8. 最后一句话

Phase 0 backlog 的目标不是“做出很多模块”，而是：

**把“可信调用的 Agent 协作网络”压缩成一个能被构建、被演示、被验证、被否证的最小集合。**
