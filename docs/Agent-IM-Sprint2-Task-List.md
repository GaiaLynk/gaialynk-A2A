# Agent IM Sprint 2 任务清单

> 基于 `Agent-IM-Phase0-Sprint-Plan.md` 的 `Sprint 2：可信调用最小闭环` 拆解。  
> 目标：让平台从“能调 Agent”升级为“能可信地调 Agent”。  
> Sprint 假设：**2 周**，核心目的是建立 Phase 0 的最小可信调用骨架。

- 总计划：`Agent-IM-Phase0-Sprint-Plan.md`

- 下一阶段：`Agent-IM-Sprint3-Task-List.md`

---

## 1. Sprint 目标

Sprint 2 结束时，必须能演示：

1. 一次普通调用被正常放行
2. 一次高风险调用被拦截、限制或要求确认
3. 两次调用都留下关键审计事件
4. 至少一个关键事件生成基础收据

如果做不到这一点，Sprint 2 视为未完成。

---

## 2. Sprint 2 明确不做

为了避免范围膨胀，本 Sprint 不做：

- 完整信誉画像
- 复杂评分系统
- 复杂风控算法
- TEE / attestation 正式落地
- 完整多维 Trust Engine
- 第三方认证与厂商认证体系
- 复杂治理工作流

一句话：**只做规则信任，不做成熟信任网络。**

---

## 3. Sprint 2 Definition of Done

以下条件全部满足，才算 Sprint 2 Done：

- Agent 有稳定身份字段
- Agent 有基础能力声明与风险等级字段
- 至少一条高风险规则可生效
- 普通调用与高风险调用会被区别处理
- 关键调用可写入审计
- 至少一个关键调用可生成基础收据
- 最终 demo 可以讲清楚“为什么这个平台比普通 Agent 工具更可信”

## 5. 详细任务拆解

## A. 对齐与规则边界

### S2-01 明确 Sprint 2 验收口径与高风险样例

- **负责人**：PM
- **目标**：统一团队对“高风险调用”最小样例的理解
- **建议样例**：
  - 普通调用：低风险文本处理或查询
  - 高风险调用：模拟一个需要确认的动作，如文件删除、敏感数据访问、设备控制
- **产出**：
  - Sprint 2 验收 checklist
  - 普通调用样例 1 个
  - 高风险调用样例 1 个
- **完成标准**：
  - 团队确认什么叫“普通调用”
  - 团队确认什么叫“高风险调用”

---

## B. 身份、来源、能力、风险四个基础字段

### S2-02 补齐 Agent 稳定身份字段

- **负责人**：后端 / 平台
- **目标**：让每个 Agent 在可信调用链里有稳定身份
- **至少包含**：
  - agent_id
  - owner / subject
  - source_type
  - status
- **完成标准**：
  - 每个 Agent 在目录和调用链中可被稳定识别

### S2-03 定义来源标识结构

- **负责人**：后端 / 平台
- **目标**：区分官方托管 / 自建 / 节点 / 厂商设备
- **完成标准**：
  - 调用链中可拿到来源标识

### S2-04 补齐基础能力声明字段

- **负责人**：后端 / 平台
- **目标**：能力不再只是自由文本描述
- **至少包含**：
  - capability_name 或 capability_key
  - 能力描述
  - 基础输入说明
- **完成标准**：
  - 至少一个 Agent 有结构化能力声明

### S2-05 补齐风险等级字段

- **负责人**：后端 / 平台
- **目标**：支持调用前风险判定
- **建议最小范围**：
  - `low`
  - `high`
- **完成标准**：
  - 至少一个能力被标记为高风险

---

## C. 最小 Trust Policy v0

### S2-06 定义最小 Trust Policy 规则集 v0

- **负责人**：PM + 后端 / 平台
- **目标**：先形成规则信任基线
- **建议规则至少包含**：
  - 身份不可验证：拒绝
  - 能力未声明：拒绝
  - 高风险能力：不自动执行
  - 来源未知且高风险：拒绝或要求确认
- **完成标准**：
  - 输出一版规则表
  - 团队能用它解释两类调用的不同决策

### S2-07 在调用链中接入 Trust Policy 判断

- **负责人**：后端 / 协议
- **目标**：让规则真正影响调用链
- **完成标准**：
  - 普通调用能放行
  - 高风险调用能被拦截/限制/要求确认

---

## D. 审计与收据

### S2-08 定义审计事件模型

- **负责人**：后端 / 平台
- **目标**：记录可信调用关键链路
- **至少覆盖**：
  - 调用发起
  - Trust Policy 决策
  - 调用放行 / 拒绝 / 确认
  - 结果回写
- **完成标准**：
  - 可查询至少一条关键调用事件链

### S2-09 写入关键调用审计事件

- **负责人**：后端 / 协议
- **目标**：将可信调用判断和结果写入审计
- **完成标准**：
  - 两类调用（普通 / 高风险）都留下审计痕迹

### S2-10 定义基础收据模型

- **负责人**：后端 / 平台
- **目标**：为关键事件生成最小可验证对象
- **建议最小字段**：
  - receipt_id
  - event_type
  - actor
  - timestamp
  - request_ref / result_ref
- **完成标准**：
  - 有一版基础 receipt 数据结构

### S2-11 关键事件生成基础收据

- **负责人**：后端 / 协议
- **目标**：让收据从模型变成实际产物
- **完成标准**：
  - 至少一个高风险相关事件能生成收据

---

## E. 最小可见体验

### S2-12 高风险确认交互最小方案

- **负责人**：前端 / PM
- **优先级**：P1
- **目标**：让高风险调用不是黑盒拦截
- **完成标准**：
  - 至少一种确认形态可用
  - 用户能看到为什么需要确认

### S2-13 来源与风险提示最小展示

- **负责人**：前端
- **优先级**：P1
- **完成标准**：
  - 可在 UI 或控制台中看到来源标签
  - 可看到高风险提示

说明：

- 如果资源不够，可先不做完整 UI，只要 demo 中可展示即可。

## F. 验证与收尾

### S2-14 普通调用 vs 高风险调用 demo 验证

- **负责人**：QA / 全员
- **目标**：确保 Sprint 2 的价值可被演示
- **完成标准**：
  - 普通调用路径稳定
  - 高风险调用路径稳定

  - Sprint 3 的节点 / 硬件前置条件列表

## 8. Sprint 2 Kill Criteria

若到 Sprint 2 末仍出现以下情况之一，应考虑立即收缩，而不是继续上更复杂的信任体系：

1. 普通调用与高风险调用无法被明确区分
2. 高风险调用只能靠人工临时判断，没有系统规则
3. 审计事件无法完整落地
4. 收据模型存在，但无法生成真实收据
5. 团队想通过引入评分系统来掩盖基础规则缺失

---

## 9. 下一步如何衔接 Sprint 3

Sprint 2 完成后，Sprint 3 应直接使用这些产出：

- Agent 身份与来源结构
- 能力与风险字段
- 最小 Trust Policy
- 审计事件模型
- 基础收据能力

然后进入：

- 节点注册 / 心跳 / 目录同步
- 硬件绑定 / 在线 / 基础接入

这样 Sprint 3 才能证明“这不仅是一个会话产品，而是一个可信网络”。

---

## 10. 一句话总结

Sprint 2 的任务清单不是为了做出完整信任体系，而是为了先证明一句话：

**这个平台知道哪些调用可以直接做，哪些调用不能轻易做。**
