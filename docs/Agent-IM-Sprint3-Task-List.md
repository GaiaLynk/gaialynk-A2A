# Agent IM Sprint 3 任务清单

> 基于 `Agent-IM-Phase0-Sprint-Plan.md` 的 `Sprint 3：网络最小证明与控制台增强` 拆解。  
> 目标：证明产品不是单机工具（若做节点则体现为网络雏形），并让控制台足以支撑 Sprint 4 的 demo 与验证。**不包含硬件。**  
> Sprint 假设：**2 周**，软件平台先行。

## 1. Sprint 目标

Sprint 3 结束时，必须能演示：

1. 一条以软件平台为主的端到端路径：会话 → Agent → 可信调用 → 审计/收据
2. 控制台可支撑上述演示，无需纯脚本
3. （若做节点）一个节点成功注册到主网，主网目录中出现节点同步来的 Agent

**不要求**：硬件绑定、硬件在线、硬件接入。

---

## 2. Sprint 3 明确不做

本 Sprint 不做：

- 硬件绑定、设备注册、在线状态、硬件接入会话
- 硬件品类验证访谈
- 完整跨节点消息中继
- 节点收益 / 分成、节点复杂治理
- 节点与硬件的复杂 UI 管理台

一句话：**只做软件网络雏形（可选）与控制台增强，硬件后置。**

---

## 3. Sprint 3 Definition of Done

以下条件全部满足，才算 Sprint 3 Done：

- 端到端软件闭环（会话 + Agent + 可信调用 + 审计/收据）可稳定演示
- 最小控制台具备：会话页、目录页、审计事件视图；来源/风险标签与至少一种高风险确认体验
- （若做节点）节点可完成注册、心跳、目录同步；主网目录中可看到节点 Agent
- 能跑通一条“主网 + Agent + 可信调用”的演示路径（+ 可选节点）

## 5. 详细任务拆解

## A. 对齐与 demo 准备

### S3-01 明确 Sprint 3 验收口径与 demo 骨架（软件为主）

- **负责人**：PM
- **目标**：团队明确本 Sprint 以软件平台可演示、控制台可支撑验收为主；节点为可选
- **产出**：Sprint 3 验收 checklist、demo 步骤草稿（无硬件）
- **完成标准**：团队对“软件闭环 + 控制台 + 可选节点”无歧义

---

## B. 控制台与可演示化

### S3-02 最小控制台：会话页可演示版

- **负责人**：前端
- **完成标准**：可创建/查看会话、查看消息流、发送消息

### S3-03 最小控制台：目录页可演示版

- **负责人**：前端
- **完成标准**：可查看 Agent 列表、将 Agent 加入会话

### S3-04 审计事件最小视图

- **负责人**：前端 + 后端
- **完成标准**：至少能查看关键调用事件

### S3-05 来源与风险标签最小展示

- **负责人**：前端
- **完成标准**：Agent 来源可见、高风险标签可见

### S3-06 高风险确认交互补齐

- **负责人**：前端 + PM
- **优先级**：P1
- **完成标准**：至少一类高风险调用有清晰确认体验

---

## C. 节点最小闭环（可选）

### S3-07 定义节点基础模型（若做节点）

- **负责人**：后端 / 平台
- **至少包含**：node_id, name, source_type, status, last_heartbeat_at
- **完成标准**：节点可在平台中被稳定识别

### S3-08 节点注册接口（若做节点）

- **负责人**：协议 / 节点 / 后端
- **完成标准**：节点可完成注册并拿到 `node_id`

### S3-09 节点心跳与状态维护（若做节点）

- **负责人**：协议 / 节点 / 后端
- **完成标准**：节点可上报心跳，平台可判断在线/离线

### S3-10 节点 Agent 目录同步最小版（若做节点）

- **负责人**：协议 / 节点 / 后端
- **完成标准**：节点至少能同步 1 个 Agent 到主网

### S3-11 主网接收并展示节点同步 Agent（若做节点）

- **负责人**：后端 / 平台
- **完成标准**：主网目录中可看到节点 Agent，带来源/节点信息

### S3-12 节点状态最小可视化（若做节点）

- **负责人**：前端
- **优先级**：P2
- **完成标准**：演示时可见节点在线状态

---

## D. 验证与收尾

### S3-13 端到端 demo 验证（软件 + 可选节点）

- **负责人**：QA / 全员
- **目标**：验证“主网 + Agent + 可信调用 + 审计/收据”路径可跑通；（若做节点）可展示节点接入
- **完成标准**：demo 至少稳定跑通 3 次

### S3-14 Sprint 回顾与 Sprint 4 输入整理

- **负责人**：PM
- **产出**：Sprint 3 结果总结、Sprint 4 demo/可视化/验证输入

- 状态：todo / doing / blocked / done

---

## 8. Sprint 3 Kill Criteria

若到 Sprint 3 末仍出现以下情况，应考虑收缩：

1. 端到端软件 demo 无法稳定跑通
2. 控制台无法支撑 Sprint 4 验收演示
3. （若做节点）节点注册后仍无法形成最小目录同步

**不**以硬件为 Kill 条件（硬件已后置）。

---

## 9. 下一步如何衔接 Sprint 4

Sprint 4 重点：完善 demo 脚本、埋点、用户访谈、Phase 0 验收与 go/no-go 判断。不新增硬件交付。

---

## 10. 一句话总结

Sprint 3 的任务清单是为了回答：**软件平台能否稳定可演示、控制台能否支撑验收；若做节点，这是不是一个网络。** 硬件不纳入本 Sprint。
