# Agent IM Phase 0 Sprint 计划（4 Sprint）

> 基于 `Agent-IM-Phase0-Backlog.md` 进一步排成按周推进的版本。  
> 假设节奏：**4 个 Sprint / 每个 Sprint 2 周 / 总计约 8 周**。  
> 目标：让团队知道“每两周最核心的交付是什么”，并避免范围再次膨胀。

## 1. 规划原则

### 1.1 核心原则

- **先跑通主闭环，再加可信，再做网络（可选）与可演示化，不做硬件**
- **每个 Sprint 都必须对最终端到端 demo 有贡献**
- **不允许为了“看起来完整”而牺牲主闭环**

### 1.2 本计划默认前提

- 主网 Hub 为主线
- Phase 0 只追求最小可信调用闭环（**软件平台先行**）
- 节点为可选增强，硬件**不纳入 Phase 0**
- 多 Agent 编排、完整声誉系统、硬件控制全部后置

---

## 2. 4 Sprint 总览

| Sprint | 主题 | 核心目标 | 关键判断问题 |
|--------|------|----------|--------------|
| **Sprint 1** | 主闭环骨架 | 会话、目录、A2A 最小跑通 | 用户能不能把 Agent 拉进会话并看到结果？ |
| **Sprint 2** | 可信调用最小闭环 | 风险识别、拦截、审计、收据 | 平台能不能证明“这不是普通 Agent 聊天工具”？ |
| **Sprint 3** | 网络最小证明与控制台增强 | Node–Hub 注册同步（可选）、控制台与可演示化 | 若做节点：这是不是一个网络？控制台能否支撑 demo？ |
| **Sprint 4** | Demo、可视化与验证 | 最终 demo、埋点、访谈、验收 | 这个方向是否值得进入 Phase 1？ |

## 3. Sprint 1：主闭环骨架

### 3.1 目标

搭出最小产品骨架，证明用户可以：

- 创建会话
- 从目录选 Agent
- 发起一次调用
- 在会话里看到 Agent 返回

### 3.2 主要范围

来自 backlog：

- `E1` 会话与消息闭环
- `E2` Agent 目录与接入
- `E3` A2A 调用映射（最小版）

### 3.3 建议完成项

- `E1-1` `E1-2` `E1-3` `E1-4` `E1-5` `E1-6`
- `E2-1` `E2-2` `E2-3` `E2-4` `E2-5` `E2-6`
- `E3-1` `E3-2` `E3-3`

### 3.4 Sprint 产出

- 最小 conversation / participant / message 数据模型
- 会话 CRUD 最小接口
- Agent 注册与列表 API
- 从目录加入会话
- 用户消息触发 A2A 请求
- A2A 响应写回会话

### 3.5 Sprint 退出标准

必须可以演示：

1. 创建会话
2. 选择一个 Agent 加入
3. 发一条消息
4. Agent 返回一条结果

### 3.6 风险提醒

- 不要在这个 Sprint 引入线程、@、复杂 UI、复杂筛选
- 不要同时做多 Agent 并行编排

### 3.7 对应执行文档

- `Agent-IM-Sprint1-Task-List.md`
- `Agent-IM-Phase0-Owner-Mapping.md`

## 4. Sprint 2：可信调用最小闭环

### 4.1 目标

让平台从“能调 Agent”升级为“能可信地调 Agent”。

### 4.2 主要范围

来自 backlog：

- `E4` 最小可信调用与审计
- `E8` 的最小必要部分

### 4.3 建议完成项

- `E4-1` Agent 稳定身份字段
- `E4-2` 高风险能力标识字段
- `E4-3` 最小 Trust Policy v0
- `E4-4` 审计事件模型
- `E4-5` 基础收据模型
- 若有余力：`E4-6`、`E8-2`

### 4.4 Sprint 产出

- Agent 来源与身份最小结构
- 基础能力声明与风险等级
- 至少一条高风险调用规则
- 关键调用写入审计
- 基础收据生成能力

### 4.5 Sprint 退出标准

必须可以演示：

1. 一次普通调用被正常放行
2. 一次高风险调用被拦截、限制或要求确认
3. 两次调用都留下审计事件
4. 至少一个关键事件生成基础收据

### 4.6 风险提醒

- 这一步只做 **规则信任**，不要试图同时做完整信誉画像
- 不要引入复杂评分系统

### 4.7 对应执行文档

- `Agent-IM-Sprint2-Task-List.md`
- `Agent-IM-Phase0-Owner-Mapping.md`

## 5. Sprint 3：网络最小证明与控制台增强

### 5.1 目标

证明产品不是单机工具（若做节点则体现为网络雏形）；同时让控制台与可演示化足以支撑 Sprint 4 的 demo 与验证。**不包含硬件**。

### 5.2 主要范围

来自 backlog：

- `E5` Node–Hub 最小闭环（可选）
- `E7` 基础控制台 / 极简 UI
- `E8` 来源/风险可视化与确认增强（部分）

### 5.3 建议完成项

- 若做节点：`E5-1` `E5-2` `E5-3` `E5-4`；若有余力：`E5-5`
- `E7-1` `E7-2` `E7-3`（E7-4 仅当做了 E5 且有余力）
- `E8-1` `E8-2` `E8-3`（部分可能已在 Sprint 2 完成）

### 5.4 Sprint 产出

- （若做节点）节点注册与心跳、节点最小目录同步
- 可演示的最小控制台（会话、目录、审计视图）
- 来源/风险标签与高风险确认的最小体验

### 5.5 Sprint 退出标准

必须可以演示：

1. 一条以软件平台为主的端到端路径：会话 → Agent → 可信调用 → 审计/收据
2. 控制台可支撑上述演示，无需纯脚本
3. （若做节点）一个节点注册到主网，主网目录中出现节点同步来的 Agent

**不要求**：硬件绑定、硬件在线、硬件接入。

### 5.6 风险提醒

- 节点阶段不要强上完整跨节点消息中继
- 硬件不纳入本 Sprint；不提前设计硬件专属 UI

### 5.7 对应执行文档

- `Agent-IM-Sprint3-Task-List.md`
- `Agent-IM-Phase0-Owner-Mapping.md`

### 6.2 主要范围

- `E7` 基础控制台 / 极简 UI

- `E9` Demo、埋点与验证

### 6.3 建议完成项

- `E9-1` `E9-2` `E9-3` `E9-4`

### 6.4 Sprint 产出

- 可演示的最小控制台

### 6.6 风险提醒

## 7. 建议的每 Sprint 验收节奏

### Sprint 1 结束时

问一个问题：

**用户和 Agent 能不能在一条会话里完成一次真实交互？**

### Sprint 2 结束时

问一个问题：

**系统能不能明确区分普通调用和高风险调用？**

### Sprint 3 结束时

问：

- （若做节点）这是不是一个网络？
- 控制台能否稳定支撑 demo 与验收？

**不追问**硬件是否为 Agent 设备（后置）。

### Sprint 4 结束时

问最终问题：

**这个“可信调用的 Agent 协作网络”值不值得继续放大？**

## 8. 建议的负责人处理方式

现在已经补了一份轻量责任映射文档，可以直接配合本计划使用：

- `Agent-IM-Phase0-Owner-Mapping.md`

你刚刚问的“第 1 个也需要怎么处理”，我的建议仍然成立：

### 不要一开始单独做一份很重的“组织架构文档”

更合适的做法是**跟着 Sprint 计划轻量挂负责人**，先按角色维度分：

| 角色 | 主要负责范围 |
|------|--------------|
| **产品 / PM** | PRD、验收标准、优先级、用户访谈、北极星定义 |
| **后端 / 平台** | 会话模型、目录、A2A、审计、收据、Trust Policy |
| **前端 / 客户端** | 会话页、目录页、审计视图、确认交互、基础控制台 |
| **协议 / 节点** | Node–Hub 协议、节点注册、目录同步（若做节点） |
| **设计 / UX** | 最小控制台、风险提示、确认流程 |
| **测试 / QA** | 端到端 demo 验证、回归、验收 checklist |

**硬件 / 固件**：Phase 0 不设主责；待 Phase 1 及之后硬件团队就位再纳入。

### 推荐下一步

`Agent-IM-Phase0-Owner-Mapping.md` 不需要很长，只需要做到：

- 每个 Epic 一个主责任角色
- 每个 Sprint 一个 owner
- 跨角色依赖清楚

现在这份文档已经补齐，和 `Sprint 1 ~ Sprint 4` 任务清单一起看会更实用。
