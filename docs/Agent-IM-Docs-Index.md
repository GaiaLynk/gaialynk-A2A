# Agent IM 文档总览首页

> 用途：作为 `Agent IM` 项目文档的统一入口。  
> 目标：帮助不同角色快速找到“现在该看哪份文档”，避免在多份规划、PRD、backlog、sprint 文档之间来回切换。
>
> **当前执行主线**：**软件平台先行**；Phase 0 只验证可信调用的软件闭环（会话、目录、A2A、审计/收据、控制台、demo）。节点为可选增强；**硬件为后续拓展轨**，待团队就位后在 Phase 1 及之后推进。

## 1. 先看这段

如果你只想知道：

- **这个项目长期想做什么**：看 `Agent-IM-Product-Plan.md`

## 5. 如果你现在要开工

最实用的打开方式不是从长文档开始，而是：

1. 先看 `Agent-IM-One-Page-Strategy.md`
2. 再看 `Agent-IM-Phase0-Sprint-Plan.md`
3. 再看 `Agent-IM-Phase0-Product-Form.md`
4. 然后只看当前 Sprint 的 `Task List`
5. 遇到范围或验收争议时，回看 `Agent-IM-Phase0-PRD.md`
6. 遇到优先级争议时，回看 `Agent-IM-Phase0-Backlog.md`
7. 遇到分工争议时，回看 `Agent-IM-Phase0-Owner-Mapping.md`

一句话：

**开工优先看“当前阶段文档”，不要一上来先读完整长期规划。**

## 6. 如果你现在要判断项目值不值得继续

建议只看这 4 份：

1. `Agent-IM-One-Page-Strategy.md`
2. `Agent-IM-Phase0-PRD.md`
3. `Agent-IM-Phase0-Backlog.md`

## 7. 当前推荐的文档使用规则

为了避免后续文档越来越多、越来越乱，建议遵守下面规则：

- `Product Plan`：只放中长期判断，不承接日常排期
- `One-Page Strategy`：只放当前阶段的战略收敛结论
- `PRD`：只放范围、用户、验收，不写任务级拆解
- `Backlog`：只放优先级、依赖、交付包，不写太细执行过程
- `Sprint Plan`：只放阶段节奏与退出标准
- `Phase0 Product Form`：只回答主产品形态、信息架构与官网/控制台关系
- `Task List`：只放具体执行项、DoD、Kill Criteria
- `Owner Mapping`：只放责任边界，不代替 sprint 管理
- `Pitch`：只服务表达，不承接细节设计

## 8. 建议的维护顺序

如果后续要更新文档，建议按下面顺序更新，而不是随手改：

1. 先改 `Agent-IM-One-Page-Strategy.md`
2. 再改 `Agent-IM-Phase0-PRD.md`
3. 再改 `Agent-IM-Phase0-Product-Form.md`
4. 再改 `Agent-IM-Phase0-Backlog.md`
5. 再改 `Agent-IM-Phase0-Sprint-Plan.md`
6. 最后改对应 Sprint `Task List`

如果涉及长期方向变化，再补改：

- `Agent-IM-Product-Plan.md`
- `Agent-IM-5-Minute-Pitch.md`

这样可以减少“上游没改、下游先漂移”的问题。

## 9. 当前这套文档最适合做什么

这套文档当前最适合支撑的，不是长期大规模组织化交付，而是：

- Founder / 小团队阶段的方向收敛
- Phase 0 的快速验证
- 团队对齐与开工
- demo、访谈与 go/no-go 判断

也就是说，它现在更像一套 **MVP 级项目操作系统**，而不是成熟公司的重流程文档库。

## 10. 一句话总结

如果只记住一句话，那就是：

**看方向，从 `One-Page Strategy` 进；看执行，从 `Sprint Plan` 进；看全局，再回 `Product Plan`。**
