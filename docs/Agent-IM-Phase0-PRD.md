# Agent IM Phase 0 详细 PRD

> 基于 `Agent-IM-One-Page-Strategy.md` 的收敛版执行文档。  
> 本文只服务 **Phase 0（0～2 月）**：验证“可信调用的 Agent 协作网络”是否成立。  
> 不追求完整平台，不覆盖中长期所有赛道。

## 1. PRD 目标

### 1.1 一句话目标

在 0～2 个月内，交付一个可运行的 **Agent 协作网络最小闭环（软件平台先行）**：

- 用户能创建会话
- 从目录中选择 Agent 加入
- 发起真实调用
- 平台完成最小可信调用控制
- 关键事件可被审计与留证
- （可选）自建节点可以连接主网
- 自有硬件接入与硬件品类验证**后置**至 Phase 1 及之后

### 1.2 本阶段要验证的核心假设

1. 用户愿意在同一会话里与多个 Agent 协作，而不是只用单点聊天。
2. “可信调用”会提升用户对平台调用 Agent 的意愿。
3. （若做节点）自建节点愿意为“发现 + 跨节点协作”连接主网。

硬件 Agent 是否被理解为“Agent 原生设备”留待 Phase 1 及之后、硬件团队就位后再验证。

### 1.3 本阶段不验证的内容

- 完整消费者增长模型
- 大规模第三方硬件生态
- **自有硬件接入与硬件品类验证**（后置至 Phase 1 及之后）
- 完整模板市场
- 多桥接 IM 平台
- 完整企业私有化
- 复杂分成与商业化体系
- 完整 IoT southbound 大范围兼容

## 2. 目标用户

### 2.1 第一阶段核心用户

**开发者型小团队 / 早期企业创新团队**

这些用户具备以下特征：

- 已经在使用多个 AI Agent、工作流工具或自动化工具
- 对“多 Agent 协作”有直观需求
- 愿意容忍早期产品粗糙度
- 对“可信调用、审计、确认”有明确价值感知

### 2.2 非目标用户

本阶段不重点面向：

- 普通消费者大众市场
- 重度智能家居玩家
- 大型企业采购与私有化团队
- 追求完整生态与大量模板的开发者社区

## 3. 核心场景

### 3.1 场景定义

用户在一个会话中引入一个或多个 Agent，围绕一个明确任务协作；平台作为中间层负责：

- Agent 发现与加入
- 调用映射
- 高风险动作拦截/确认
- 结果回写
- 审计与收据

### 3.2 最小闭环示例

1. 用户创建一个会话
2. 从目录中选择 `Agent A` 与 `Agent B`
3. 用户输入任务，如“帮我总结这份资料并给出行动建议”
4. 平台将消息映射给目标 Agent
5. 若调用中涉及高风险能力，则触发确认或拦截
6. Agent 返回结果，平台写入消息流
7. 平台记录关键审计与收据事件

### 3.3 Phase 0 不纳入的复杂场景

- 多步骤自动编排
- 多设备协同控制
- 跨多个 IM 平台的复杂桥接
- 完整 southbound 执行 Agent 大规模接入

## 4. 产品范围

### 4.1 必须交付

#### A. 会话与消息闭环

- 创建/查看会话
- 人与 Agent 同处一个会话
- 基础消息流
- A2A 请求/响应映射到会话消息

#### B. 最小目录

- Agent 注册
- Agent 查询/列表
- 从目录加入会话
- `agent_type` 区分：`logical` / `execution`

#### C. 最小可信调用

- Agent 稳定身份
- 来源标识：官方托管 / 自建 / 连接节点 / 厂商设备
- 基础能力声明
- 高风险调用默认不自动执行
- 最小审计日志
- 基础收据

#### D. Node–Hub 最小闭环（可选）

- 节点注册
- 心跳
- 目录同步 MVP

### 4.2 应该交付

- 基础可见的来源/信任标识
- 基础风险等级字段
- 至少一种高风险确认交互
- 基础控制台 / 可演示 UI（支撑 demo 与验收）

### 4.3 明确不做

- 完整线程
- @ 路由增强
- 完整意图路由
- 复杂编排 DSL
- 多维信誉画像完整版
- 大量 Connector
- 多设备协同场景
- 大规模第三方执行 Agent 生态
- **自有硬件接入与硬件品类验证**（后置）

### 4.4 后续扩展（不计入 Phase 0 交付）

- **硬件最小闭环**：首次绑定、设备注册、在线状态、基础会话接入
- **硬件品类验证**：验证用户是否把它理解为 Agent 原生设备，而不是“小主机”
- **推进时机**：待硬件团队就位后，在 Phase 1 及之后推进

## 5. 用户故事

### 5.1 用户故事（P0）

1. 作为用户，我可以创建一个会话，并看到自己和 Agent 的消息都在同一条对话流中。
2. 作为用户，我可以从目录中选择一个 Agent 加入会话，而不需要知道底层部署细节。
3. 作为用户，我发起一次任务后，平台能把任务转发给正确的 Agent，并把结果回写到会话。
4. 作为平台管理员，我可以看到关键调用的审计记录和关键收据。
5. 作为高风险调用的执行方，平台能在必要时阻止自动执行，或要求用户确认。
6. （若做节点）作为节点运营者，我可以把节点连接到主网并同步最小目录。

### 5.2 用户故事（P1，若有余力）

1. 作为用户，我能看到 Agent 的基础来源与风险提示。
2. 作为用户，我能对高风险调用完成一次明确确认。
3. 作为管理员，我能区分不同来源 Agent 的调用记录。

## 6. 功能模块说明

### 6.1 会话模块

**必须有**：

- Conversation 列表与详情
- Message/Event 顺序流
- 人 / Agent 参与者模型

**成功标准**：

- 用户可完成一次完整会话
- Agent 返回内容可自然展示在会话中

### 6.2 目录模块

**必须有**：

- Agent 注册数据结构
- 列表/详情
- 加入会话动作

**最小展示字段建议**：

- 名称
- Agent 类型
- 来源标识
- 基础能力描述
- 风险提示（如有）

### 6.3 可信调用模块

**必须有**：

- 稳定身份
- 来源标识
- 基础能力声明
- 高风险能力默认不自动执行
- 最小审计日志
- 基础收据

**Phase 0 推荐规则基线**：

- 未验证身份：不放行
- 未声明能力：不放行
- 高风险能力：默认不自动执行
- 关键事件：必须写入审计

### 6.4 节点模块（可选）

**若做节点，必须有**：

- 节点注册
- 心跳
- 节点状态可见
- 目录同步 MVP

**Phase 0 不要求**：

- 跨节点中继完整闭环
- 节点收益分成

### 6.5 硬件模块（后置）

硬件接入（首次绑定、设备注册、在线状态、会话接入）**不纳入 Phase 0**，待硬件团队就位后在 Phase 1 及之后推进。长期验收重点仍为：用户是否把硬件理解成 Agent 设备，而不是普通主机。

---

## 7. 交互与体验要求

### 7.1 用户侧体验要求

- 会话与 Agent 调用的路径足够短
- 不要求用户理解 A2A、节点、协议这些底层概念
- 高风险调用提示必须明确、可理解

### 7.2 管理与审计体验要求

- 能按会话看到关键事件
- 能区分普通消息和关键调用事件
- 能看到高风险调用为何被拦截或要求确认

### 7.3 硬件体验要求（后置）

Phase 0 不交付硬件；待 Phase 1 及之后硬件团队就位，再定义：首次绑定步骤、插上即在线、状态反馈机制等。

---

## 8. 数据与审计要求

### 8.1 最小数据对象

- conversation
- participant
- message
- agent
- node（若做节点）
- audit_event
- receipt

device 留待 Phase 1 及之后硬件接入时引入。

### 8.2 最小审计对象

至少记录以下事件：

- 会话创建
- Agent 加入会话
- Agent 调用发起
- 调用被放行 / 限制 / 拒绝
- 高风险调用确认或拦截
- 结果写回
- （若做节点）节点注册 / 心跳异常

---

## 9. 成功指标

### 9.1 北极星

**每周可信调用完成数**

### 9.2 配套指标

- 每周活跃会话数
- 多 Agent 会话占比
- 首次会话成功率
- 高风险调用被正确拦截/升级比例
- 关键调用收据覆盖率
- 审计事件覆盖率
- （若做节点）已连接节点数

### 9.3 定性验证指标

- 用户是否说“这个平台让我更放心地调 Agent”
- 用户是否理解高风险控制的价值
- （若做节点）节点是否愿意连接主网

---

## 10. 验收标准

### 10.1 产品验收

1. 用户可创建会话并加入至少一个 Agent
2. Agent 可完成一次真实调用并回写结果
3. 高风险调用至少有一种确认/拦截机制
4. 关键事件有最小审计与收据
5. （若做节点）一个自建节点可完成注册与目录同步

### 10.2 Demo 验收

必须可演示一条端到端流程：

- 主网用户发起会话
- 选择一个 Agent
- 完成一次调用
- 出现一次可信调用判断
- 留下审计/收据
- （若做节点）可同时展示一个连接节点接入主网

### 10.3 品类验收

至少 3～5 位目标用户在体验后能复述：

- 这是一个“可信调用的 Agent 协作网络”
- 高风险控制让自己更放心调用 Agent

硬件品类验证后置至 Phase 1 及之后。

## 11. 风险与缓解

### 风险 1：范围再次膨胀

**缓解**：所有新增需求必须回答一句话：  
“它是否直接增强 Phase 0 的可信调用闭环？”

### 风险 2：信任体系过度设计

**缓解**：Phase 0 只做规则信任，不做复杂声誉系统。

### 风险 3：硬件分散研发资源

**缓解**：Phase 0 采用软件平台先行；硬件接入与品类验证后置至 Phase 1 及之后，待硬件团队就位后再推进。

### 风险 4：节点能力过早复杂化

**缓解**：Phase 0 只做节点注册、心跳、目录同步，不强求完整跨节点中继。

---

## 12. 里程碑建议

### 里程碑 1：主网最小会话闭环

- 会话可创建
- Agent 可加入
- 基础消息流可跑通

### 里程碑 2：可信调用最小闭环

- 来源标识
- 基础能力声明
- 高风险默认拦截
- 审计与收据

### 里程碑 3：节点最小闭环（可选）

- 节点注册
- 心跳
- 目录同步

### 里程碑 4：硬件最小闭环（后置）

硬件绑定、在线、会话接入留待 Phase 1 及之后、硬件团队就位后推进。

---

## 13. 最终判断问题

Phase 0 结束时，只回答一个核心问题：

**用户是否会因为“可信调用”而选择在这个网络里协作 Agent，而不是继续分散地用单点 Agent 工具？**

如果答案是“会”，Phase 1 再继续放大：

- 信誉画像
- 编排
- 本地行动中枢
- 一键部署
- 更多节点与生态

如果答案是“不会”，则应先收缩定位，而不是继续加功能。
